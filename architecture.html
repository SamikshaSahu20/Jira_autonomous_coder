Autonomous Coder

From Jira ticket to reviewed, running code — fully autonomous. Two annotated diagrams mapping every layer.

DIAGRAM 01
Jira → AI Generation Pipeline
Background daemon polls Jira, detects In Progress tickets, and runs them through a three-agent AI chain that writes, reviews, and commits working source code.
Jira / External Python Service AI / LLM File / Storage
Jira Cloud Sprint Board — autonomous-coder
Source of all work items. Status lifecycle: To Do → In Progress → In Review → Done. Ticket summary, ADF-formatted description, and acceptance criteria are fetched via Jira REST API v3 & Agile 1.0. Board ID set in .env.
polls every N minutes
Daemonwatcher.py
Polling loop. Detects sprint changes, calls run_pipeline() for any In Progress story. Persists state to .watcher_state.json to skip duplicates.
→
Orchestratorpipeline.py
Iterates over In Progress tickets. For each: fetches full details, calls generate_code(), posts Jira comment with file summary, then transitions ticket status.
Jira Clientjira_reader.py
Basic Auth. Parses ADF (Atlassian Document Format) to plain text. Posts generation-complete comment. Transitions ticket via POST /rest/api/3/issue/{key}/transitions.
+
Stack Detectorcontext_analyzer.py
Scans output/ for existing code to detect project stack (e.g. javascript/express). Caches as .stack_cache_*.json to avoid repeated LLM calls.
enriched spec + stack context
Code Engine
generate.py — Three-Agent Orchestrator
Drives the sequential AI chain. Reads .agents/skills/frontend-design/SKILL.md at runtime and injects design guidelines into Agent 2. Parses all code blocks from LLM responses, extracts filenames from first-line comments. Safety fallback: if Agent 3 returns fewer files than Agent 2, missing files are merged back in automatically.
AGENT 01
Analyst
Transforms vague Jira ticket + stack context into a precise development spec with file plan, architecture, and acceptance rules.

Tester
Reviews and fixes all bugs. Upgrades weak designs. Writes test cases. Generates Mermaid flowchart. Re-outputs all corrected files.

Web Application & Frontend Architecture
JWT-secured React dashboard that lets reviewers browse generated apps, preview live Node servers in an iframe, and chat with the code agent.
Browser → React SPA → axios + JWT → FastAPI :8000 → manifest / output/ → Node.js :dynamic → iframe preview
Client Layer — React 19 + Vite + Tailwind + Recharts + Mermaid.js

Auth Gate

KPI + Table

5-Tab View

axios instance. Interceptor auto-injects Authorization: Bearer <token> on every request.

API Layer — FastAPI (uvicorn :8000) — serves React /dist as static SPA
POST /api/login
Auth
sha256_crypt hash check against users.json. Returns signed HS256 JWT via python-jose.
GET /api/tickets
Ticket List
Reads manifest.json, groups by ticket key, aggregates file counts, language totals & timings.
POST /run
Run App
Fixes package.json deps, runs npm install, patches server.js, starts Node via subprocess.Popen. Returns preview URL.
POST /chat
Ask Agent
Passes code context + question to call_llm(agent="coder"). Returns explanation or fix suggestions.

Data & Storage Layer
manifest.json
File Registry
67+ entries. Each: ticket, filename, language, run_dir, created_at. Append-only.
users.json
Auth Store
Managed by create_user.py CLI. Stores username + sha256_crypt hash. No database required.
output/ASK-XXX/
Source Files
One flat dir per ticket. All JS/HTML/CSS source, flowchart MD, metrics, timings, run scripts.
.env
Config
Portkey key, per-agent model/provider, Jira credentials, WEBAPP_SECRET, board ID, sprint name.
↓ subprocess.Popen · dynamic port · RUNNING_APPS dict ↓
Preview Layer — Dynamic Node.js Servers (one per ticket, on demand)
runtime
Node Process
One process per ticket. Started on demand. Tracked in RUNNING_APPS. Restarted if dead on retry.
auto-install
npm install
Skipped if node_modules/ exists. Auto-injects missing 3rd-party deps into package.json from a safe allow-list.
auto-patch
_patch_server_js
Removes deprecated Mongoose options. Rewrites blocking connect().then(listen) to non-blocking. Adds cors(). Runs once.
fallback
Static Fallback
If Node crashes at startup, falls back to python -m http.server to at least serve the index.html.
